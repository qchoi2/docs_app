# DESIGN_INTEGRATION — getdesign.md 기반 디자인 자산 활용 지침
_2026-07-08 · UI 구현 전에 반드시 확인할 디자인 연동 절차._

## 1. 목적

웹 UI를 구현할 때 임의의 새 디자인 시스템을 만들지 않고, 사용자가 제공하는 `getdesign.md`와 그 문서가 참조하는 디자인 파일을 우선 활용한다.

## 2. 구현 전 필수 절차

UI 구현 Step에 들어가기 전에 코딩 모델은 다음을 수행한다.

```text
1. 저장소 루트 또는 사용자 지정 위치의 getdesign.md를 찾는다.
2. getdesign.md가 참조하는 디자인 파일, 이미지, 토큰, CSS, 컴포넌트, 스크린샷을 확인한다.
3. 사용할 수 있는 자산을 DESIGN_AUDIT.md에 요약한다.
4. UI_PRODUCT_SPEC.md의 화면 요구사항과 디자인 자산 사이의 충돌을 정리한다.
5. 충돌이 없으면 getdesign.md의 색상·폰트·간격·컴포넌트 스타일을 우선 적용한다.
```

## 3. 금지사항

- getdesign.md를 확인하지 않고 UI 스타일을 임의로 만들지 않는다.
- 사용 가능한 디자인 토큰이 있는데 Tailwind/Bootstrap 등 새 디자인 체계를 임의 도입하지 않는다.
- 기능 구현보다 장식적 애니메이션을 우선하지 않는다.
- 디자인 파일이 없다는 이유로 UI 구현 전체를 중단하지 않는다. 이 경우 기능 우선 MVP를 만들고 스타일은 최소화한다.

## 4. DESIGN_AUDIT.md에 포함할 내용

```text
# DESIGN_AUDIT

## 확인한 디자인 소스
- getdesign.md 경로:
- 참조 파일:

## 사용 가능한 토큰
- 색상:
- 폰트:
- 간격:
- radius/shadow:

## 사용 가능한 컴포넌트
- 버튼:
- 입력창:
- 카드:
- 테이블:
- 배지:
- 사이드바:

## UI_PRODUCT_SPEC 적용 계획
- 검색 화면:
- 결과 카드:
- 문단 뷰어:
- 히스토리 사이드바:
- 대시보드:

## 충돌/부족한 부분
- ...
```

## 5. 우선순위

1. 기능적 명확성
2. 근거 확인 가능성
3. getdesign.md 준수
4. 반응형/시각적 polish

계약서 검색앱이므로 화려한 UI보다 결과 신뢰와 문단 확인이 중요하다.

## 6. UI 구현 명령 예시

```text
UI 구현 전에 getdesign.md를 읽고 DESIGN_AUDIT.md를 작성하라.
getdesign.md가 제공하는 색상, 폰트, 간격, 카드/버튼 스타일을 우선 사용하라.
디자인 자산이 없거나 불완전하면 기능 우선 MVP로 구현하되, DESIGN_AUDIT.md에 부족한 항목을 기록하라.
```
