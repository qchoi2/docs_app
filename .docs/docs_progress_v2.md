# M&A 계약서 검색 프로그램 — 진행 문서 v2

_최종 업데이트: 2026-07-08_
_v1 대비 변경: 티어 사다리를 "벡터 지향"에서 "구조화 메타데이터 지향"으로 전환.
개발·운영 전 과정에서 Claude Code(구독제, 비용 0)를 배치 작업자로 최대 활용하는
원칙을 명문화._

## 0. 설계 원칙 (v2에서 확정)

1. **검색 축과 답변 축을 분리한다.** (v1 유지)
2. **진짜 병목은 검색이 아니라 정독(컨텍스트)이다.**
   후보를 넓게 잡아 원문을 통째로 읽히는 구조는 컨텍스트 한계에 부딪히거나
   에이전트의 추측을 유발한다. 해법은 검색 개선이 아니라
   **문서를 미리 구조화해 원문 정독량 자체를 줄이는 것**.
3. **이 코퍼스에서 임베딩의 기대 효용은 낮다.**
   계약서는 보일러플레이트 비율이 높아 청크 임베딩이 유형별로 뭉치며,
   실제로 궁금한 미세 차이(상한 금액, 특이 조항)를 벡터 유사도가 잡지 못한다.
   어휘가 유한한 도메인이므로 **조항 태그 + 용어사전 + FTS5**가
   임베딩보다 싸고 정확할 가능성이 높다. 임베딩은 최후 수단(T4)으로 격하.
4. **AI-fill 원칙: 사람이 수작업으로 만들 뻔한 모든 산출물의 초안은
   Claude Code 배치가 생성한다.** 사람은 검수자 역할만 한다.
   구독제이므로 배치 호출 비용은 0. 대상 목록은 §5 참조.
5. **지침이 아니라 도구로 제약한다.** "추측 금지" 같은 소프트 지침 대신,
   부분 읽기 도구·file_key 인용 강제 등 구조적 장치로 답변 품질을 보장.
6. **모든 아키텍처 결정은 골든 질의 세트의 재현율로 판정한다.** 감(感) 금지.
7. **API 사용 원칙 (2026-07-08 사용자 승인): "작업자는 Claude Code,
   판정기·폴백은 API".** 대량 작업(전량 추출, 사전 초안 등)은 구독제
   Claude Code가 수행한다. API는 다음 세 경우에만, 각 실행이 명확히 저렴할 때
   사용한다: (a) **단발성 판정** — 분류 폴백, 시범 추출 교차검증
   (Haiku/Sonnet, 1회성), (b) **T1/T2 런타임 경량 폴백** — Claude Code 세션
   없이 CLI에서 즉답이 필요할 때 Haiku 요약 (회당 $0.20 미만 상한, 실측
   기준 회당 수 센트 예상), (c) T4의 G2 본답변. 모든 API 경로는
   호출 전 예상 비용 로그를 남긴다.

## 1. 요청사항 (사용자) — v1과 동일

- 수백 개의 M&A 계약서 샘플을 DB화, 자연어 질의로 리서치하는 개인용 프로그램
- 구독제 Claude Code를 AI 역할로 우선 활용 (API 비용 부담)
- 저렴 → 고품질 스펙트럼을 단계적으로 선택 가능하게
- RAG + API 방식도 옵션으로 유지

### 확인된 데이터 특성 — v2.2 갱신
- **규모 확정 (2026-07-08): 2,245개 파일, 총 855MB. 단기 증가분 10~20% 예상**
  - 파일당 평균 ~380KB → 추출 텍스트 전량 대략 수천만~1억 토큰 규모로 추정
    (색인 리포트의 char_count 합계로 확정할 것)
  - **함의 1**: 계층 1.5 전량 추출은 "하룻밤"이 아니라 **수일 분할 배치**.
    **Claude Code Pro 플랜 확정(2026-07-08)** — Pro의 사용한도(시간창)를
    감안하면 전량은 1~2주에 걸친 분할이 현실적. 우선순위 큐가 그래서
    중요: **SPA → SHA → SSA → MOU** (사용자 지정) → ATA/BTA →
    JVA·공동투자 → CB·BW·EB → 주식교환 → 분할합병·분할계획서 → 기타
    (잔여 순서는 실무 활용 빈도 추정에 따른 Claude 판단, 변경 자유)
  - **함의 2**: 전량을 API로 대체하면 Haiku 기준으로도 수십 달러 단위 —
    "작업자는 Claude Code" 원칙이 규모 때문에 더 중요해짐
  - **함의 3 (실행 환경 확정)**: **스크립트는 PC에서 실행 (RAM 16GB)**,
    NAS는 원본 저장소. 색인 산출물(cs_index)은 PC 로컬 디스크에 두고
    원본만 네트워크로 읽음. 16GB면 T4의 bge-m3 임베딩·리랭커도 PC에서
    무리 없음 (NAS CPU 제약 소멸)
  - **함의 4**: zip 유래 중복 가능성 × 2,245건 → dedup의 실효 가치 상승
- 주로 .docx, 일부 .pdf, .hwp 없음
- 폴더가 `유형_언어`로 기분류 (예: `05-1_CB_인수계약_국문`)
- 계약유형: SPA, SSA, SHA, ATA/BTA, CB·BW·EB, 분할계획서, 분할합병,
  MOU, 공동투자, JVA, 주식교환
- 언어: 국문/영문/국영문
- `.zip` 존재 (`CPG_계약_선례 (1)~(6)`) → **색인 대상에서 제외** (사용자가 추후
  제거 예정). 색인기는 zip을 건드리지 않고 "제외됨" 리포트만 남김
- **당사자명 대부분 실존 명칭 잔존** → doc_meta에 **실명 + 역할 병기**로 저장
  (아래 §7 결정 참조). "○○사가 매수인인 계약" 류 질의 지원
- `99_MnA_계약_외_Sample_Docs` 포함 여부 추후 결정

## 2. 수정된 티어 사다리 (저렴 → 고품질)

| 티어 | 검색 | 답변 | 비용 | 상태 |
|---|---|---|---|---|
| **T1** | catalog 메타 필터(유형·언어) | G0 목록만 | 0 | ⬜ 구현 대상(본 패키지는 지시서) |
| **T2** | + FTS5 전문검색 + 용어사전 확장 | G1 Claude Code | 0 | 🔶 grep판 구현, FTS5·사전 미착수 |
| T2+ (옵션) | T2와 동일 | **G1.5 Haiku 경답변** — Claude Code 세션 없이 CLI 즉답 | 회당 ~2¢ (상한 20¢) | ⬜ Phase 1 |
| **T3** | + 조항 태그 검색(doc_meta) | G1 Claude Code — **조항맵 우선, 원문 최소 정독** | 0 (배치 1회) | ⬜ |
| **T4** | + 벡터 하이브리드 + 재순위 | G2 API 모델 선택(Haiku~Opus) | 유료(답변만) | ⬜ **확정 — §3-T4 상세 설계** |

- v1의 "T3 = 벡터"를 폐기. T3의 실체는 **구조화 메타데이터**.
- **T4는 구현 확정** (사용자 결정, 2026-07-08). 다만 순서는 유지한다 —
  T3 안정화 후 착수하며, 벡터·재순위 등 각 구성요소는 골든 세트
  **절제 실험(ablation)으로 기여도를 측정해 채택/폐기를 개별 판정**한다.
  구현하되 맹신하지 않는다는 원칙.

### 하이브리드 검색 흐름 (v2)
1. **1차**: 메타 필터 (유형·언어·버전·중복그룹)
2. **2차**: FTS5 + 용어사전 확장 검색, 또는 doc_meta 조항 태그 SQL 질의
3. **판독**: 조항 맵/요약을 먼저 읽고, 필요한 문서·조항만
   `read_contract.py`로 **부분 정독**

핵심 효과: "earn-out 있는 SPA의 손해배상 상한 비교" 같은 질의가
본문 검색이 아니라 `WHERE ctype='SPA' AND has_earnout=1` + 조항맵 열람으로
환원됨. 원문 정독은 최종 소수 문서의 해당 조항으로 국한.

## 3. 계층 구조 (v2)

### 계층 1 — 공통 토대 (구현됨, 보강 필요)
**`index_contracts.py`**
- 기존: 폴더 스캔 → txt 캐시 → catalog.sqlite, 유형·언어 자동 분류, 증분 실행
- **보강 사항 (v2 신규)**:
  - `content_hash` 컬럼 추가 → **중복 그룹 리포트** (zip 해제 후 필수)
  - 파일명 휴리스틱 태깅: `is_draft`, `version_hint`
    (v2/최종/수정/draft/execution 등 패턴)
  - **표(테이블) 셀 추출 확인** — 당사자·대금은 본문이 아니라 표·별지에
    있는 경우가 많음. 각주·머리글/바닥글은 best-effort 처리 방침 명시
  - 리포트에 **미지원 확장자 목록** 추가 (.doc 구형식 등 잠복 가능)
  - **txt 캐시에 문단 마커 [¶n] 삽입** — 계층 1.5의 location 좌표와
    read_contract.py 부분 읽기의 선행 요건
  - FTS5 가상 테이블 생성 (trigram 토크나이저 또는 공백 정규화 전처리)

catalog 스키마 (v2):
```
file_key, path, folder, filename, ctype, lang, ext, size,
txt_path, char_count, status, indexed_at,
content_hash, dup_group, is_draft, version_hint
```

### 계층 1.5 — 보강(enrichment) 패스 ★ v2 신설, 핵심 투자
**`enrich_contracts.py` + 추출 프롬프트** — Claude Code 배치로 실행
- 계약서를 1건씩 읽혀 구조화 JSON 추출 → `doc_meta` 테이블 저장
- **규모 대응 (2,245건 기준)**:
  - **재개형 배치**: 문서 단위 커밋 — 언제 중단돼도 다음 실행이 이어감
    (증분 skip과 동일 메커니즘). 진행률 파일(`enrich_progress.json`) 유지
  - **우선순위 큐 (확정)**: **SPA → SHA → SSA → MOU** (사용자 지정) →
    ATA/BTA → JVA·공동투자 → CB·BW·EB → 주식교환 → 분할합병·분할계획서
    → 기타. `enrich_contracts.py --priority` 기본값으로 내장하되 덮어쓰기 가능
  - **사용한도 인지**: Claude Code 한도 도달 시 배치가 우아하게 멈추고
    재개 시각 안내. 전량 완료까지 수일 소요를 전제로 일정 계획
  - 중복(dup_group) 대표 1건만 추출 — 규모를 실질적으로 줄이는 첫 수단
- 증분 실행 (이미 추출된 건 skip)
- **추출 스키마 버저닝**: `meta_schema_version` 컬럼 → 프롬프트 개선 시
  재추출 대상을 식별 가능

doc_meta 스키마 (초안):
```
file_key (FK), meta_schema_version, extracted_at,
parties_json,            -- 당사자 (익명화된 샘플이면 역할만)
deal_type_detail,        -- 자동분류 세분화 (예: 구주/신주, 국내/크로스보더)
consideration_json,      -- 대금 구조 (현금/주식/earn-out 여부, 조정 메커니즘)
clause_map_json,         -- 조항 존재 맵 + 각 조항의 위치(문단 범위) + 3줄 요약
  -- 대상 조항: 진술보장 / 선행조건 / 확약 / 손해배상(상한·존속기간 포함) /
  --            해제 / 경업금지 / earn-out / MAC / 분쟁해결(중재/재판) 등
special_notes,           -- 특이사항 한 줄
definitions_json,        -- 정의 조항의 핵심 정의어 목록
confidence,              -- 추출 신뢰도 (low/med/high, 자가 평가)
```
- `clause_map_json`의 **위치 정보**가 `read_contract.py --section`의
  좌표가 됨 → 부분 읽기의 기반

### T2 도구 계층
- **`search_contracts.py`** (기존) — FTS5 백엔드 교체, 용어사전 자동 확장,
  `--json` 출력에 `file_key` 필수 포함
- **`read_contract.py`** (신규) — `--file-key X --section 손해배상`
  부분 읽기. 원문 전체 cat 금지의 구조적 대안
- **`term_dict.sqlite` / yaml** (신규) — 국·영문 조항 용어 동의어 사전
  (예: 진술보장 = 진술 및 보증 = R&W = Representations and Warranties)
- **`CLAUDE.md`** (개정) — 워크플로우를 다음 순서로 고정:
  1. 메타/조항태그 질의 → 2. 조항맵·요약 열람 → 3. 필요 시에만 부분 정독
  - **모든 주장에 file_key 인용 필수** (검증 가능 형식 강제)
  - 후보 폭증 시 "축소 → 조항맵 → 부분정독" 3단 압축

### T1/T2 확정 스펙 ★ 구현 확정 (수용 기준 포함)

**`search_contracts.py` v2 스펙**
- 백엔드: catalog 메타 질의 + **FTS5(trigram)** 본문 검색
- 옵션: `--type --lang --limit --exclude-drafts --json --context N --expand strict|normal|broad`
  - `--kw`: term_dict 동의어 **자동 확장** (끄기: `--no-expand`)
  - `--dedup`(기본 on): 같은 dup_group에서 대표 1건 + "동일 문서 n곳" 표시
  - [T3 예약] `--clause 태그 [--present/--absent]`
- **정확 일치 우선 랭킹**: 사용자가 친 원어 매치를 동의어 확장 매치보다
  상위에 (원질의/확장질의 결과를 RRF로 융합 — 확장으로 인한 오탐이
  상위를 오염시키지 않게). T4 융합 코드의 축소판으로 미리 구현
- `--json` 필수 필드: file_key, path, ctype, lang, is_draft, dup_group,
  matched_terms(어떤 변이가 어디[¶n]서 맞았는지), snippet
- 본문 정규화 전처리(색인 시): NFC 유니코드 정규화, 전각→반각,
  하이픈 변이 통일(earn-out/earnout), 공백 정규화

**`answer_quick.py` (G1.5, 신규 소품)**
- `search_contracts.py --json` 상위 후보(기본 10건)의 스니펫만 Haiku에
  전달해 2~3문장 즉답 + file_key 인용. 용도: Claude Code 세션을 열
  것도 없는 가벼운 질의, 폰/원격에서의 빠른 확인
- 가드레일: 입력 상한 20k 토큰, **호출 전 예상 비용 출력**, 회당
  $0.20 초과 예상 시 실행 거부(분할 안내). 실측 예상 회당 1~3¢
- G1.5는 "요약 폴백"이지 판독기가 아님 — 비교·정독형 질의는
  Claude Code(G1)로 안내 문구 출력

**수용 기준 (골든 세트 연동)**
- T1 완료 = Q01~Q04 전부 통과 (개수 정확, dedup, 드래프트 구분)
- T2 완료 = Q05~Q08 통과 (용어 변이·언어 교차 포착) + Q19·Q20·Q21
  (무정답 인정, 중복 함정, 범위 이탈 거절)
- 통과 판정은 `eval_search.py` 실행 결과로만 (재색인 후 자동 재실행 —
  회귀 테스트 겸용)

### T4 계층 — 벡터 하이브리드 + API 답변 ★ 구현 확정 (상세 설계)

**핵심 아이디어: T3의 산출물이 T4의 입력이 된다.**
조항 단위 청킹을 별도로 개발하지 않고, **clause_map의 location(¶범위)을
그대로 청크 경계로 사용**한다. 이로써 (a) 청킹 품질 문제가 사라지고
(b) 모든 청크가 조항 태그·file_key로 라벨된 상태로 임베딩되어
벡터 검색 결과가 곧바로 부분 정독 좌표가 된다. 보일러플레이트 뭉침
문제(설계원칙 3)도 조항 태그로 층화 검색(같은 태그 내에서만 유사도 비교)
하여 완화한다.

**검색 축 (R2) 구성**
- **청킹**: clause_map location 기반 조항 청크 + 커버되지 않는 구간은
  ¶ 마커 기준 슬라이딩 윈도우(600~800자, 20% 겹침) 폴백.
  청크 메타: file_key, canonical_tag, ¶범위, lang
- **임베딩**: `bge-m3` 로컬 (다국어 — 국·영문 혼재 코퍼스에 적합,
  dense+sparse 동시 지원). 후보: `multilingual-e5-large`.
  선정은 골든 세트 소표본으로 A/B 후 확정
- **벡터 저장**: **LanceDB** 권장 (파일 기반·서버리스 → NAS 운영에 유리,
  별도 데몬 불필요). 대안 Chroma. catalog.sqlite와 file_key로 조인
- **하이브리드 융합**: FTS5(BM25) 결과 + 벡터 결과를 **RRF(Reciprocal
  Rank Fusion)**로 결합 — 점수 정규화 없이 견고, 구현 단순
- **재순위**: `bge-reranker-v2-m3` 로컬 크로스인코더. 융합 상위 ~50 청크
  → 상위 10으로 압축. CPU에서도 50건이면 수 초 내
- **질의 임베딩 경로**: 질의를 그대로 임베딩하되, CLAUDE.md 질의 해석
  단계가 만든 canonical 태그를 층화 필터로 병용

**답변 축 (G2) 구성**
- **모델 라우팅**: Haiku(단건 요약·목록 정리) / Sonnet(기본 — 비교·분석) /
  Opus(정밀 법률 뉘앙스 비교, 다문서 횡단). 기본값 Sonnet, 사용자 토글
- **비용 가드레일** (설계 불변식):
  1. 후보 축소·재순위는 전부 로컬(무료) — API에는 **최종 청크만** 전달
  2. 질의당 입력 상한(예: 조항 청크 10개, ~15k 토큰) — 초과 시 분할 제안
  3. 실행 전 예상 비용 표시(모델×토큰), 사용자 승인 후 호출
  4. 동일 질의+동일 후보 조합은 답변 캐시 재사용
- **인용 불변식 유지**: G2에서도 file_key + verbatim 인용 강제
  (프롬프트에 청크 메타를 함께 전달)

**평가 (ablation 필수)**
골든 세트를 4구성으로 측정: ① FTS5+사전만 ② +벡터(RRF) ③ +재순위
④ +층화필터. 재현율 개선이 없는 구성요소는 파이프라인에서 제거한다.
특히 ②가 ① 대비 개선이 미미하면 벡터는 "의미 질의 전용 폴백"으로 강등.

### 평가 계층 ★ v2 신설
- **`golden_queries.yaml`** — 골든 질의 20~30개: 실제 물어볼 법한 질의 +
  기대 문서 목록(file_key) 쌍
- **`eval_search.py`** — 골든 세트로 재현율/정밀도 측정.
  티어 업그레이드 전후로 실행 → T4(임베딩) 착수 여부를 데이터로 판정

  "grep/태그로 못 잡는 질의 유형" 식별 근거

## 4. 스펙트럼 요약: 무엇이 품질을 올리고 무엇이 돈이 드는가

| 투자 | 비용 | 기대 효과 |
|---|---|---|
| FTS5 + 용어사전 | 0 (반나절) | 표현 변이 커버, 랭킹, 속도 |
| 조항 추출 배치 (계층 1.5) | 0 (하룻밤 배치) | 질의의 SQL화, 정독량 격감, 비교 질의 가능 |
| 부분 읽기 도구 | 0 (소규모) | 컨텍스트 절약, 답변 근거 정밀화 |
| 골든 세트 + eval | 0 (반나절) | 모든 이후 결정의 근거 |
| 임베딩 하이브리드 + rerank (T4) | 0~ (로컬 연산·복잡도) | **구현 확정** — 구성요소별 ablation으로 채택 판정 |
| API (Haiku~Opus) | 유료 | 최종 답변 품질 상한, 원할 때만 토글 |

### API 공통 가드레일 ★ 모든 API 경로(A9·A10·G1.5·G2)에 강제 적용

**예산은 사용자가 입력한다** — `api_budget.yaml` 한 곳에서 관리
(동봉 파일에 입력란 준비됨. **회당·실행당 2개 상한만 입력**하면 되고,
비어 있으면 모든 API 도구가 실행을 거부):
```yaml
per_call_limit_usd: null    # 1회 호출 상한 (권장 0.20)
per_run_limit_usd:  null    # 1회 실행(질의/배치) 총 상한 (권장 2.00)
```
- 모든 API 도구는 실행 전 **견적(대상 건수 × 예상 토큰 × 단가)을 출력**하고,
  상한 초과 예상 시 호출하지 않는다. 모델 문자열·단가도 이 파일에서 관리.
- **지출 원장** `api_ledger.jsonl`: 모든 호출의 시각·용도·모델·토큰·비용을
  기록 (차단 기준은 아니고 추적용 — 누적 지출은 사용자가 원장으로 확인).

**루프·낭비 방지 불변식**:
1. **단방향 큐 원칙**: 모든 배치는 "대상 목록 확정 → 소진 → 종료" 구조.
   결과를 보고 스스로 대상을 다시 만드는 자동 루프 금지 — 재실행은
   항상 사람의 명시적 명령으로만.
2. **재시도 상한 2회** (지수 백오프). 2회 실패한 건은 `failed` 마킹 후
   건너뛰고 배치 끝에 실패 목록 보고 — 한 문서에 무한 재시도 금지.
3. **입력 해시 캐시**: (프롬프트+입력) 해시가 같으면 재호출 없이 캐시 반환.
   같은 문서·같은 질의에 두 번 돈 쓰지 않음. G2 답변 캐시도 이 메커니즘.
4. **입력 절단 상한**: 호출당 입력 토큰 상한(용도별 설정, 예: A10은 2k,
   G1.5는 20k)을 코드에서 강제 — 프롬프트 실수로 원문 전체가 딸려가는
   사고 방지.
5. **에이전트의 자율 호출 금지**: Claude Code(G1)가 작업 중 스스로 유료
   API를 부르지 않는다. G1.5/G2는 사용자가 해당 도구를 직접 실행할 때만.

### A2 품질 관리 루프 (중요)
```
추출 프롬프트 v1 → 10건 시범 추출 → 사람 검수(정확도 확인)
→ 프롬프트 수정 → 전량 배치 → confidence=low 건만 재검토
→ 스키마/프롬프트 개선 시 meta_schema_version 올리고 해당 버전만 재추출
```
- 신뢰도 자가평가(`confidence`)를 넣어 사람 검수 대상을 low로 좁힘
- 추출 실패·비정형 문서는 `status`로 표시하고 검색에서 제외하지 않되 표기

## 6.5 증분 갱신 런북 — 계약서 추가·변경·삭제 시 절차 ★ 신설

설계 전제: 모든 계층이 증분형(색인 skip, 추출 skip, 임베딩 신규분만)이므로
갱신은 "전체 재구축"이 아니라 **변경분만 흘려보내는 파이프라인**이다.

### A. 파일 배치 규칙 (사용자)
- 기존 `유형_언어` 폴더 규칙에 맞춰 넣는다 (분류가 공짜로 되는 경로).
- 새 유형/언어 폴더를 신설하면 type_rules.yaml에 패턴 1줄을 먼저 추가.
- zip으로 넣지 않는다 (색인 제외됨).

### B. 일상 추가 (수십 건 단위) — 소요 수 분
1. `index_contracts.py` 실행 (증분) → 리포트의 **"이번 실행 변경분"** 확인:
   신규 / 내용변경 / 이동 / 삭제(missing) / 미분류 / 신규 중복그룹
2. 미분류 발생 시: type_rules 보강 또는 [AI-A10] 폴더 단위 판정 → 재색인
3. [T3 가동 후] `enrich_contracts.py` 실행 — 대상은 자동 선정:
   **doc_meta 없는 신규분 + txt_hash 불일치(stale)분**. 우선순위 큐 적용
4. [T4 가동 후] 임베딩 증분 — 신규·변경 file_key의 청크만
5. `eval_search.py` 실행 — 기존 통과 문항이 깨지지 않았는지 회귀 확인
- Phase 2 이후에는 3~5단계를 **`update_contracts.py` 한 명령**으로 체인
  실행 (각 단계는 개별 실행도 가능하게 유지).

### C. 대량 추가 (수백 건+, 덤프 병합) — B에 더해
6. **중복 리포트 정밀 확인** — 덤프는 기존 문서와의 중복 확률이 높음.
   dup_group 병합 결과를 훑고, 유사·비동일 쌍은 [AI-A5] 버전 판정
7. **[AI-A11] 사전 보강 재실행** — 신규 문서군의 표기 관행 수집
8. **골든 세트 재점검** — expected_filter 기반 문항은 자동으로 유효하나,
   expected_files 고정 문항은 신규 문서가 정답 집합을 바꿀 수 있음 →
   eval이 "정답 후보 변동 가능" 플래그를 띄우고 사용자가 갱신 승인

### D. 삭제·수정 파일의 처리 규칙 (시스템 동작, 자동)
- **삭제**: 경로가 사라진 레코드는 지우지 않고 `status='missing'`으로
  전환 — 검색에서 제외되고, 복원 시 status만 되돌아옴 (doc_meta 보존).
- **수정**: 같은 경로의 내용이 바뀌면 바이트 해시가 달라져 **새 file_key**가
  생긴다. 옛 레코드는 자동으로 missing(대체됨) 처리하고 리포트에
  "내용변경"으로 표기. 새 file_key는 신규 문서로 추출·임베딩 큐에 진입.
- **stale doc_meta 탐지**: doc_meta.txt_hash ≠ 현재 content_hash 인 문서는
  위치좌표(¶)를 신뢰할 수 없음 → 리포트에 "재추출 필요 n건", 재추출
  전까지 답변 시 조항맵을 인용하되 좌표 기반 부분 정독은 원문 검색으로 폴백.

## 7. 유의점 (v2 갱신)


- **전량 추출은 수일짜리 일정이다 (2,245건).** 우선순위 큐로 자주 쓰는
  유형부터 T3 혜택을 받게 하고, dup 대표 추출로 실질 건수를 먼저 줄여라.
  Claude Code 한도 도달은 오류가 아니라 예정된 일시정지다.
- **조항 추출의 정확도가 시스템 전체의 상한이 됨.** A2 품질 루프를 생략하지
  말 것. 잘못된 태그는 "못 찾음"보다 나쁜 "틀리게 찾음"을 만든다.
- **추출 결과는 원문의 대체물이 아니라 색인이다.** 최종 답변의 근거는 항상
  원문 부분 정독으로 확인 (CLAUDE.md에 명시).
- **당사자명 처리 (결정됨)**: 실명이 대부분 남아있으므로 doc_meta의
  parties_json에 **실명과 역할(매도인/매수인/대상회사/투자자 등)을 병기**.
  실명 질의("○○사 계약")와 역할 질의("매수인 측 진술보장") 모두 지원.
  실명이 없는(익명화된) 샘플은 역할만 기록하고 `name: null`로 일관 처리.
- **후보 폭증**: 3단 압축(축소 → 조항맵 → 부분정독)을 도구 차원에서 강제.
- **스캔 PDF (결정 프로세스 확정)**: 색인 리포트의 `empty` 건수를 본 뒤
  사용자가 OCR 투자 여부를 결정한다. 그 전까지 empty 문서는 "본문 검색
  불가"로 답변에 고지만 한다.
- **`99_MnA_계약_외_Sample_Docs` (결정)**: 기본 제외, `--include-misc`
  옵션으로 포함 선택 (type_rules의 exclude_by_default 플래그로 구현).
- **API 예산 미입력 시**: api_budget.yaml의 상한이 비어 있으면 모든 API
  도구는 실행을 거부하고 입력을 안내한다 (기본값으로 몰래 동작 금지).

## 2026-07-08 업데이트 메모 — 파일럿 운용 및 구현 보완 반영

본 문서는 로드맵과 과거 진행 메모를 포함한다. 실제 Phase 0~1 구현 기준은
`IMPLEMENTATION_BRIEF.md`가 우선한다. 특히 `doc_meta`의 최종 Phase 1 DDL은
Brief §2.4의 `json` 통합 저장 스키마를 따른다. 이 문서 중 `parties_json`,
`consideration_json`, `clause_map_json` 등 컬럼 분리형 설명은 Phase 2 설계 논의의
구버전 메모로만 본다.

### 추가된 운영 요구사항
1. **파일럿→전체 확장**: 처음에는 일부 계약서 subset으로 성능·검색 품질을
   검증하고, 만족 시 같은 루트 하위에 나머지 파일을 추가해 증분 색인한다.
   임시 복사 폴더로 파일럿을 돌린 경우 전체 운영용 `cs_index`는 새로 만든다.
2. **검색 설명 가능성**: 검색 결과 JSON에 `why`, `score_breakdown`,
   `snippet_paras`를 포함한다.
3. **운영 디버깅 도구**: `inspect_file.py`, `open_text.py`, `plan_extract.py`를
   Phase 1 구현 대상에 포함한다.
4. **수동 보정**: `data/manual_overrides.yaml`을 로드해 자동분류 오류를 코드 수정 없이
   보정한다.
5. **실패 원인 분류**: Brief §2.4.1의 enum 기준으로 `error_reason`을 저장하고 리포트에 encrypted_pdf,
   corrupt_docx, pdf_text_empty, permission_denied 등으로 집계한다.
6. **중복 대표 선정**: non-draft → final/signed/clean → 짧은 경로 → 읽기 쉬운
   filename → file_key 순으로 대표본을 선택한다. `--show-duplicates`로 펼쳐본다.
7. **로그 분리**: `query_log.jsonl`은 검색 실행 로그, `agent_log.jsonl`은 최종 답변 로그로 분리해 term_dict/golden_queries 보강에 쓴다.

### 파일럿 완료 판단 기준
- 파일럿 문서 100~300건에서 색인 실패 원인 리포트가 해석 가능할 것
- 자주 쓸 검색 10~20개가 기대 문서를 상위권에 노출할 것
- 단일 검색이 3초 안팎으로 끝날 것
- 미분류/언어 오분류/드래프트 오분류를 `manual_overrides.yaml`로 보정 가능할 것
- 이 기준을 만족하면 전체 문서를 추가하고 `eval_search.py`를 다시 실행한다.
