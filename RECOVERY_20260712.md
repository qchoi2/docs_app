# 복구 안내 — catalog.sqlite / ui_state.sqlite 손상 (2026-07-12 점검)

_이 문서는 2026-07-12 리뷰 세션에서 발견한 DB 손상과 그 복구 절차를 정리한다.
세션16까지의 코드·테스트는 정상이며, 손상은 **색인 데이터베이스 파일**에 국한된다._

## 무엇이 문제인가

`cs_index`의 SQLite 파일 3개 중 2개가 손상됐다.

| 파일 | 상태 | 근거 |
|---|---|---|
| `catalog.sqlite` (89MB) | **손상(truncated)** | `PRAGMA integrity_check` → *malformed*. 헤더의 페이지 수 146,813(≈601MB)가 유효(change_counter==version_valid_for)인데 실제 파일은 21,802페이지(89MB)뿐. 즉 데이터의 약 85%가 잘려 나감 |
| `ui_state.sqlite` (32KB) | **손상** | `integrity_check` → *malformed* |
| `jobs.sqlite` (24KB) | 정상 | `integrity_check` → *ok* (같은 경로·같은 읽기 방식으로 정상 판독 → 읽기 경로 문제가 아니라 파일 자체 손상임을 입증) |

`catalog.sqlite.corrupt_20260712.bak`(63MB)도 이미 손상 상태다 — 이전에 한 번 손상이
감지돼 백업된 흔적이며, 그 이후 재빌드한 현재 파일도 다시 손상됐다.

**영향:** 전체 코퍼스 색인(2,000여 건)과 T3 조항맵(`doc_meta`)이 현재 **읽을 수 없다.**
검색·enrich·eval이 모두 이 카탈로그에 의존하므로 복구 전까지 런타임 동작 불가.

## 왜 이런 일이 생겼나 (추정 원인)

거의 확실한 원인은 **샌드박스(리눅스)에서 FUSE로 마운트된 호스트 폴더 위의 SQLite에
대용량 쓰기를 한 것**이다. 세션16의 1,969건 enrich 일괄 + 전체 재색인처럼 큰 쓰기가
동기화/네트워크성 마운트 위에서 진행되면서 파일이 잘린 것으로 보인다. 근거:
- 최근 크게 쓰인 두 DB(`catalog` 07-12 19:27, `ui_state` 07-12 00:32)만 손상, 오래된
  `jobs.sqlite`(07-11)는 정상.
- `cs_index`에 `.fuse_hidden*` 잔재가 다수, `.git/index`도 반복적으로 손상돼 온 이력
  (progress.md 기록) — 모두 이 마운트가 잠금·부분쓰기에 취약함을 시사.

**교훈:** 무겁게 쓰는 색인/enrich 배치는 **로컬 디스크에서** 돌려야 한다. `--out`을
동기화 폴더(OneDrive/Dropbox/네트워크 드라이브)나 원격 마운트로 두고 대량 쓰기를 하지 말 것.

## 다행인 점 — 복구 재료는 온전하다

손상은 카탈로그 DB뿐이고, 그걸 다시 만들 원재료는 전부 살아 있다.

- 원본 코퍼스 `contract_docs/` : **2,244건** (docx 1467, doc 561, pdf 180, jpg 34, xlsx 1, eml 1)
- txt 추출 캐시 `cs_index/txt/` : **2,106건**
- **T3 결과 JSON `cs_index/enrich_results/` : 1,999건 (전부 schema v2, 전부 파싱 정상)**
- .doc 변환본 `cs_index/converted/` : docx 497건 + `manifest.json`(562 항목)
- `enrich_progress.json`, 색인 리포트들도 존재

핵심: `enrich_contracts.py`는 `enrich_results/<file_key>.json`이 있으면 **AI 재추론 없이
그대로 검증·적재**한다(코드 확인 완료). `file_key`는 원본 바이트 해시라 재색인 후에도 값이
같으므로, **T3 조항맵은 무료·결정적으로 복원**된다.

### 2) 카탈로그 재빌드 (원본에서 전체 재색인)

`.doc` 변환본과 매니페스트가 남아 있어 재변환 없이 재사용됩니다.

## 추가 발견 (2026-07-12 가드 작업 중)

이 손상은 DB에만 국한되지 않았습니다. **소스 파일 `webapp.py`도 작업 트리에서 잘려
있습니다**(825줄, 825번째 줄 `return ("raw", 200, ... payload,`에서 미완성 → 컴파일 불가).
다행히 **커밋된 버전(HEAD)은 온전(1167줄, 정상 컴파일)** 하므로 git에서 복구하면 됩니다.

```
git restore --source=HEAD -- webapp.py      # 또는: git checkout HEAD -- webapp.py
python -c "import py_compile; py_compile.compile('webapp.py', doraise=True); print('ok')"
```

`.git/index`가 손상 상태(webapp.py 등이 "삭제"로 표시)라 위 명령이 안 먹으면, 인덱스를
새로 만든 뒤(`git read-tree HEAD` 또는 리셋) 다시 복구하세요. 손상 원인은 DB와 동일
(마운트 위 대용량/동기화 쓰기)로 추정됩니다.

**교훈 재확인:** 이 환경(FUSE/동기화 마운트)은 쓰기 중 파일을 잘라먹습니다. 이번 세션에서도
파일 저장이 여러 번 truncate돼 append 방식으로 복구했습니다. **큰 파일 쓰기·색인·enrich는
반드시 PC 로컬 디스크에서** 하세요.
